- **`src/services/analytics/strategy.js`** — `getStrategies()`/`getActiveStrategy()`:
  несколько сохранённых стратегий живут МАССИВОМ в профиле (`userProfile.strategies` +
  `activeStrategyId`), не отдельной коллекцией Firestore. Обратная совместимость со
  старым единственным `userProfile.strategy` — синтезируется на лету при чтении.
- **`src/components/backtest/Backtest.js`** — страница (доступ через `TrustedRoute` в
  `App.js`, тот же список `TRUSTED_UIDS`, что обходит верификацию почты — НЕ через
  Firestore-роль `admin`, у пользователя роль `pro`). Выбор любой сохранённой стратегии,
  тикер/тип инструмента/лет истории, редактор правил выхода (можно временно покрутить
  без сохранения в Капитал), кнопка запуска.
  - Результаты: карточки (накопленная доходность со сложным процентом, сделки, винрейт,
    профит-фактор), SVG-кривая капитала (`EquityCurve.js`), **один график на всю

    анализ на момент входа», что в Журнале, пересчитанную НА ДАТУ ВХОДА этой конкретной

    историю до). Показывает оба периода раздельно + вердикт (зелёный если оба в плюсе,

## Реальные баги, пойманные ЖИВЫМ тестом (не сборкой/тестами) в этом этапе

1. **Маркеры сделок пропадали** — `leg.timestampUtc` из Firestore приходит объектом
   `{seconds,...}`, а не строкой; `new Date(...)` от него тихо давал `Invalid Date`.
2. **Кнопка «Подставить по стратегии» говорила «подставлено», поля оставались пустыми**
   — `formatNumber()` (ru-RU локаль, запятая вместо точки) подставлялась в
   `<input type="number">`, который тихо обнуляет невалидную строку. Исправлено на
   `.toFixed(2)`.
3. **Гонка состояний в МОЕЙ методологии тестирования** (не баг приложения) — если
   скриптом сразу после смены стратегии в выпадающем списке кликать по кнопкам правил
   выхода без паузы, React не успевает закоммитить сброс `exitRules`, и настройки от
   предыдущей стратегии протекают в следующий прогон. Урок: между `select` и
   конфигурацией exit-полей нужен отдельный вызов/пауза, не всё одним скриптом.

**Важная находка пользователя (curve-fitting, реально пойман руками):** у стратегии #2
раздвинул стоп/тейк с 2%/4% на 3%/5% → доходность на IMOEXF подскочила с +19% до +59%.

параметра, дающий огромный скачок результата, это красный флаг подгонки, а не находка.**
Именно это и стало поводом для функции out-of-sample holdout (см. выше).

## Известные упрощения/ограничения движка (см. также комментарии в коде)

- **Нет «ИЛИ» между условиями.** У пользователя в шаблонах несколько раз встречается
  «RSI ниже 55 ИЛИ MACD положительный» — наш конструктор считает только через общий %
  готовности (по сути «И» с порогом), явного «ИЛИ» между конкретной парой условий нет.
  Обходили включением обоих + подбором порога — это ПРИБЛИЖЕНИЕ, не то же самое.
  Пользователь сам пришёл к выводу, что стоит сделать настоящие группы условий с И/ИЛИ —
  **согласована как будущая фича, не начата.** Не путать со срочностью — сначала нужно
  разобраться, почему нынешние 8 стратегий плохие, это не решится одним «ИЛИ».
- **Нет реального сайзинга позиции** — весь P&L считается в % от цены входа, не в
  рублях/лотах. «Накопленная доходность» — честный сложный процент ПРИ ПОЛНОМ
  реинвестировании, не то же самое, что реальная фиксированная доля риска (2% на сделку,
  как в реальных настройках Капитала). Явно подписано в UI.
- **Уровни/фигуры на обзорном графике бэктеста считаются «как сейчас»** (на последнюю
  свечу), не пересчитываются заново на каждый день прогона — это только визуальная
  сверка «похоже ли на правду». Drill-down по конкретной сделке — честный, на дату входа.
- **Только дневной таймфрейм.** Интрадей — отдельная задача, гейтится доступностью
  данных (грант на платный MOEX ещё не решён, см. память).

- **Никогда не вводить пароли в формы** — ни при каком разрешении, даже в тестовый

- После входа пользователя — можно управлять уже авторизованной сессией через

  массового заведения стратегий через конструктор.
- Каждое изменение — `npm run test:import` (9/9) + `npm run build` → живая проверка в
  браузере → коммит с подробным сообщением (WHY, реальные находки живого теста) → `git
  push origin main`.
- Не молчать о том, что НЕ проверено вживую — писать честно.
- При скриптовой автоматизации UI (много кликов подряд) — между сменой состояния
  (например, выбор стратегии в выпадающем списке) и следующим действием, которое от неё
  зависит, нужна пауза/отдельный вызов — иначе React не успевает закоммитить, и
  получаются ложные результаты (см. «гонка состояний» выше — наступали на эти грабли
  дважды).
