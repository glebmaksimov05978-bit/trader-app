# TraderPro — Архитектура аналитического движка (v3)

Обновление v2 → v3 сделано после того, как модуль импорта Т-Банка был доведён до
рабочего состояния (2026-07). Часть предпосылок v2 писалась до этого и уже устарела —
в лучшую сторону: детерминированный PDF-парсер оказался точным (не понадобился AI как
основной путь), и позиционная модель с историей шагов (`legs`) закрывает почти весь
раздел 7.2 без единой ручной галочки.

## Принцип (без изменений)

Три уровня достоверности данных, каждый явно помечен в системе:

1. **Точные алгоритмы** (индикаторы, статистика) — 100% точность, без AI, дёшево
2. **Алгоритмы-кандидаты** (формализуемые фигуры теханализа) — оценка с вероятностью, без AI, дёшево
3. **AI-мнение** (Vision по скриншоту) — субъективная оценка, дорого, только по явному запросу

AI никогда не считает статистику. AI только формулирует текст на основе готовых чисел.

Между статистикой и AI стоит **движок выводов** — правила с порогами, которые формируют
выводы детерминированно. AI получает готовый вывод и только облекает его в человеческий
текст. Обязательное правило движка: **минимальный размер выборки**. Вывод формируется
только при n ≥ 25 сделок по анализируемому срезу; при меньшей выборке результат
помечается как «гипотеза, данных пока мало» и никогда не подаётся как уверенный факт.

## Что изменилось с v2 (важно для планирования)

- **PDF-парсер — детерминированный, не AI.** `parseMethod: 'exact'` покрывает
  практически весь реальный объём сделок; AI (`parseMethod: 'ai'`) — только фолбэк,
  когда точный парсер не справился с конкретным отчётом.
- **`averaged`, `partialClose` больше не ручные поля.** Каждая импортированная позиция
  хранит `legs` — полную историю шагов (цена, объём, комиссия, время каждой докупки/
  частичного закрытия). Усреднение против позиции, частичное закрытие и их результат
  **вычисляются из legs автоматически**, не требуют самоотчёта трейдера.
- **Новые поля, которых не было в схеме v2:**
  - `instrumentType` (`stock` | `future` | `currency`) — готовый срез для движка выводов
  - `openCommissionPerUnit` — комиссия открытия учитывается на каждом шаге позиции, не
    только при закрытии (раньше у акций из P&L вычиталась только комиссия продажи)
  - `reportPeriod` (границы дат отчёта) + sanity-предупреждения (цена/объём/дата вне
    периода) — контроль качества входных данных
  - `unparsedDealNumbers` — строки, похожие на сделку, но не распознанные парсером;
    видно пользователю в предпросмотре импорта, не теряются молча
  - коллекция `importArtifacts` (несопоставленные закрытия + РЕПО-операции) — сырьё для
    будущей кнопки «пересобрать журнал»
- **Известное ограничение движка выводов про фьючерсы:** вариационная маржа брокера за
  период ≠ P&L по позициям за тот же период, если позиция пересекает границу периода
  (плюс фандинг у вечных фьючерсов вроде IMOEXF). Сравнение «журнал vs отчёт брокера»
  корректно делать только по позициям, целиком открытым и закрытым внутри одного периода.
  Иначе движок будет уверенно врать — ровно то, что трёхуровневая система должна исключать.

## 1. Модуль хранения сделок — расширенная схема

(раздел не менялся по существу относительно v2, кроме полей выше — см. предыдущий пункт)

### Firestore коллекция `trades`

**Существующие поля:** ticker, direction, orderType, entryPrice, exitPrice, stopLoss,
takeProfit, volume, lot, commission, depositSize, depositPercent, rr, pnl, setup,
emotion, notes, status ('open'|'partial'|'closed'), source ('calculator')

**Время и идентификация:** openedAt, closedAt, durationMinutes, exchange, timeframe,
dayOfWeek, hourOfDay (МСК)

**Источник данных:** dataSource, brokerSource, parseMethod, sourceTransactionIds,
fingerprint, legs, remainingVolume, instrumentType, openCommissionPerUnit

**Исполнение:** `intendedEntryPrice` — **заполняется только при ручном создании сделки
через Калькулятор** (там трейдер и так вводит плановую цену для расчёта риска). При
дальнейшем «усыновлении» такой сделки импортом отчёта — поле сохраняется, не
затирается. Импорт сам по себе это поле никогда не создаёт: у брокера нет данных о
том, что трейдер планировал до нажатия кнопки — только факт исполнения.

**Психология:** emotionBefore, emotionAfter, confidenceLevel, selfRating (без изменений)

**Стоп/тейк (обновление 2026-07-10):** `stopLoss`, `takeProfit` были в схеме только со
стороны Калькулятора — теперь доступны и в TradeModal (журнал), необязательные поля.
Фундамент для будущего вычислимого детектора «перенос стопа» (заявленный стоп vs
фактическая цена выхода), без ручной галочки `stopMoved` из v2 (отклонена — см. раздел
7, «Отклонено»).

**Журнал изменений:** подколлекция `trades/{id}/history` (без изменений)

## 3. Движок выводов и анализ привычек (7.1 + 7.2 из v2 — теперь основной фронт работы)

Детекторы дают сырые цифры → движок выводов сравнивает их с порогом → формулирует
вывод по детерминированному правилу (n ≥ 25 обязательно). Вся цепочка без AI на этом
этапе — AI подключается только чтобы облечь готовый вывод в человеческий текст.

Все детекторы ниже считаются **из уже существующих полей, без единой ручной галочки**
от трейдера — это прямое следствие того, что `legs` и время теперь есть у каждой
импортированной сделки:

| Детектор | Источник данных | Ручной ввод нужен? |
|---|---|---|
| Асимметрия удержания (прибыль режу, убыток пересиживаю) | `durationMinutes` × знак `pnl` | нет |
| Комиссионный налог на суету | комиссии по legs / валовой P&L, по `instrumentType` | нет |
| P&L по типам инструментов | `instrumentType` | нет |
| Усреднение против позиции и его цена | `legs` (докупка при цене хуже входа) | нет |
| Тильт/отыгрыш после убытка | `openedAt` соседних сделок + `volume` | нет |
| Карта времени (часы/дни/первые 30 мин сессии) | `openedAt` в МСК | нет |
| Пересиживание до экспирации | `expiredUnclosed` | нет |
| Соблюдение дневного лимита | `dailyLossLimit` из профиля (₽) + внутренняя константа движка «3 убытка подряд» | нет |

**Уточнение про дневной лимит (решено 2026-07-10):** правило «2 стопа подряд → стоп»
из ранних черновиков — артефакт, такой настройки у пользователя никогда не было.
Реальный сигнал: сделки, открытые после превышения настроенного пользователем
`dailyLossLimit` (вкладка «Капитал»), плюс невидимая пользователю константа движка
`MAX_CONSECUTIVE_LOSSES = 3` — не настройка, в UI не выводится, упоминается только в
тексте вывода.

Ручными остаются только эмоции и сетап — то, что биржа объективно знать не может.
Дополнительно (решено 2026-07-10): в TradeModal добавлены необязательные поля
`stopLoss`/`takeProfit` (факт: где стоял стоп) — они уже были в схеме от Калькулятора,
теперь их можно указать и у любой сделки журнала. Это фундамент для будущего детектора
«перенос стопа»: сравнение заявленного стопа с фактической ценой выхода. Поля v2
`stopMoved`, `followedRules`, `checklistResults`, `tradeType`, `newsInfluence` как
ручные галочки — **отклонены** (см. «Отклонено» ниже): их никто не заполняет, а
вычислимая замена из legs/стопа покрывает ту же аналитику честнее.

**Принцип подачи — «3 привычки недели»:** движок считает все детекторы, но на
дашборд выводит только 2-3 самых денежно-значимых вывода (отсортированные по тому,
сколько рублей стоила привычка), с примером конкретной сделки. Остальное — в
раскрывающемся полном отчёте. Это решает риск перегрузки интерфейса без отказа от
расчётов.

**Окно расчёта (решено 2026-07-10):** топ-3 считаются по скользящим последним 30 дням
(`HABITS_WINDOW_DAYS`), а не по всей истории — иначе изжитая привычка из старых сделок
никогда не уйдёт из топа. Если в окне меньше 25 закрытых сделок — фолбэк на всю
историю, с явной подписью на карточке, какое окно сейчас используется. Когда окно
активно, каждый детектор дополнительно считается на сделках ДО окна (`previous`) — в
полном отчёте видно «раньше проявлялось / сейчас нет — прогресс» или «привычка новая».

**Порядок реализации** (по стоимости внедрения, от дешёвого к дорогому):
1. Асимметрия удержания, комиссионный налог, P&L по типам — считаются из готовых
   агрегатов позиции, без разбора legs
2. Усреднение, тильт/отыгрыш — требуют разбора legs и попарного сравнения соседних
   сделок по времени
3. Карта времени (включая первые 30 мин сессии), дневной лимит, пересиживание
   фьючерса до экспирации — группировка по дате/часу и разовые события

**Статус (2026-07-10): все 8 детекторов реализованы и протестированы на реальном
отчёте** — `src/services/analytics/insightsEngine.js`. Движок (`computeWeeklyHabits`)
ранжирует по costRub, режет топ-3, порог n≥25 обязателен для каждого среза отдельно
(включая под-срезы вроде конкретного часа в карте времени, а не только общую выборку).

## 4. Модуль паттерн-детекции (алгоритм-кандидат, без AI)

**Статус (2026-07-10): реализовано и протестировано на реальных сделках.**

### Источник свечей (шаг 1, индикаторы)
`src/services/marketData/candles.js` — дневные свечи, гибридный источник: если у
пользователя есть `tinkoffToken` (Настройки) и инструмент stock/future — сначала
Tinkoff `MarketDataService/GetCandles`; иначе (или для currency, под которую в
tinkoff.js пока нет резолвера инструмента) — MOEX ISS API (`iss.moex.com`, без токена).
CORS подтверждён рабочим на реальных запросах (акции `engines/stock/markets/shares`,
фьючерсы `engines/futures/markets/forts`).

`src/services/analytics/indicators.js` — RSI(14), MACD-гистограмма, дистанция до
SMA200 (%), объём к среднему за 20 дней. Точная математика, без интерпретации
хорошо/плохо — это зона движка выводов (раздел 3), не этого модуля.

### Уровни и фигуры (шаг 2)
`src/services/analytics/patterns.js`:
- **EMA9 / EMA100 / EMA200** как «плавающая» поддержка/сопротивление — для каждой
  сделки показывается: цена выше/ниже, на сколько %, растёт EMA или падает.
- **Статические уровни поддержки/сопротивления** — кластеризация свинг-точек (локальных
  экстремумов) по цене (допуск ~0.6%), уровень считается значимым от 2 касаний.
- **Двойное дно/вершина** — два свинга одного типа на близкой цене (допуск 2%) с
  откатом между ними ≥2%. Скан ограничен последними ~15 свингами — иначе год истории
  даёт десятки случайных совпадений, которые трейдер не видел вживую.
- **Пробой уровня** — цена на баре входа пересекает статический уровень; уверенность
  выше при подтверждении объёмом (>1.3× среднего).
- **Треугольник/флаг** — сужение диапазона по последним 5 свингам (максимумы вниз,
  минимумы вверх) или заметное сжатие амплитуды относительно предыдущего отрезка.
- **Упрощённая 5-волновая структура** — проверка чередования свингов + два текстбук-
  правила (3-я волна не самая короткая, 4-я не заходит в зону 1-й), направление
  последней волны. **Не полноценная теория Эллиотта** (нет коэффициентов Фибоначчи,
  нет правила чередования коррекций, нет контекста волны старшего порядка) — уверенность
  жёстко ограничена потолком ~65%, всегда показывается чеклист пройденных правил и явная
  оговорка «проверьте глазами». Пользователь может дополнить критерии после того как
  посмотрит на результат на своих сделках.

Все детекторы считаются только по свечам ДО и НА баре входа сделки — заглядывать в
будущее (после openedAt) запрещено правилом кода, иначе это не анализ, а подгонка
задним числом.

**Показ:** та же раскрывающаяся строка в Журнале, что и индикаторы (единый блок
«Технический анализ на момент входа») — один запрос свечей на сделку, кэш в
`trades/{id}.technicalAnalysis.{indicators,patterns}`, кнопка «Обновить» форсирует
пересчёт. Список фигур-кандидатов обрезан до топ-5 по уверенности.

Каждый алгоритм возвращает `{pattern, confidence}` — процент соответствия формальным
критериям, не AI-оценка. Работает на исторических ценах MOEX/Tinkoff API, без
скриншотов.

## 5. Модуль AI Vision (по явному запросу, дорого)

*Перенесено из v2, статус не менялся.*

Только когда: алгоритм не нашёл паттерн, а пользователь уверен, что он был; стратегия
на субъективном визуальном признаке; нажата кнопка «Разобрать сделку». Без скриншота —
сначала просит скриншот, токены не тратятся. Результат всегда `dataConfidence:
'ai-opinion'`, в «точную» статистику не попадает.

*Перенесено из v2, статус не менялся.*

- ⚪ «Данных недостаточно»

  PDF» (parseMethod)

`confidence: 'confirmed' | 'hypothesis'` и подписью `n=X` в полном отчёте — не значком,
но по смыслу то же самое: пользователь всегда видит, насколько крепка цифра.

### 7.3. Автодетект рыночного контекста
Без изменений относительно v2: тренд/волатильность из цен MOEX API, единственный
ручной чекбокс — «день важных новостей».

### 7.4. Аналитика проскальзывания — ОТЛОЖЕНО НАВСЕГДА (решение 2026-07-11)
Пользователь решил не строить отчёт: основной поток сделок идёт через импорт отчётов
брокера, у которых `intendedEntryPrice` не бывает в принципе — срез n≥25 «ручных»
сделок реалистично не наберётся никогда, и карточка вечно висела бы в состоянии
«гипотеза, данных мало» (мёртвый вес в интерфейсе). Плюс у лимитных заявок
проскальзывание почти нулевое по определению. Поле `intendedEntryPrice` продолжает
писаться из Калькулятора (бесплатно) — если стиль торговли сместится к планированию
каждой сделки через Калькулятор, к отчёту можно вернуться, данные будут уже накоплены.

### 7.5. Версии стратегий, 7.6. История развития трейдера
Без изменений относительно v2.

### Отклонено (решение принято, не возвращаться)
- Ручная оценка качества исполнения звёздочками — дубль selfRating + автоматики по
  проскальзыванию
- Ручные чекбоксы рыночного контекста как основной механизм — заменены автодетектом
- Ручные галочки `averaged`/`partialClose` как основной источник — заменены разбором
  `legs`, которые теперь есть у любой импортированной позиции
- **(добавлено 2026-07-10)** Ручные поля психологии/дисциплины из v2 — `stopMoved`,
  `followedRules`, `checklistResults`, `tradeType`, `newsInfluence` — их бы никто не
  заполнял регулярно. `stopMoved` заменён вычислимым сравнением `stopLoss` (теперь
  доступен в TradeModal) с фактической ценой выхода — задел под будущий детектор, сам
  детектор ещё не реализован. Остальные три поля решено не реализовывать вообще —
  ценность ниже трудозатрат на то, чтобы трейдер их действительно заполнял.
- **(добавлено 2026-07-10)** Правило «2 стопа подряд → стоп» как готовая пользовательская
  настройка — не существует, было ошибочно записано в предыдущей сессии. Реальная
  реализация — раздел 3, детектор «дневной лимит» (dailyLossLimit + скрытая константа).
